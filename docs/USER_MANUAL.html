Get the plugin.

living-manual runs inside Claude Code. Clone the repo, then point a session at it:

git clone https://github.com/dougstanford/living-manual.git
claude --plugin-dir /path/to/living-manual

In any repo you want documented, run /living-manual:manual and answer the setup questions. You get a page like this one: your product, your brand, your code. The rest of this manual explains what that page does.

Requires git and python3. Jira sync needs the Atlassian MCP.

One file that stays true.

living-manual is a Claude Code plugin. Point it at a repo and it writes a branded, interactive user's manual, then holds that manual to the code as releases ship.

The manual is one self-contained HTML file. No build step, no server, no external assets: it opens in any browser and travels with the repo. This copy lives at docs/USER_MANUAL.html and documents the plugin itself, built by the plugin.

Three parts keep it honest. A base marker records the commit the manual last matched. A pre-push guard blocks releases that would leave the manual behind. And an update flow reads only what changed since the marker and revises only the sections those changes touched.

Reading it is also writing it. Headings take notes that become developer tickets. Features with planned changes carry roadmap previews, so feedback lands on the plan instead of the past. A queue check shows what's already filed, so duplicates become additions. The rest of this manual walks each of those surfaces.

Notes become tickets.

Every heading here is a feedback surface. The manual meets you at the moment of reaction, then carries that reaction into the development loop with its context intact.

Making a note.

Hover any heading and click ✎ make a note. Pick a type: an idea for something new, feedback on how a feature behaves, or a bug. Write one line on what should change, and as much detail as you have. The placeholder text tells you what a useful report of that type contains.

Copy for Claude Code builds a structured payload: your note plus the section, its anchor, the manual version, and instructions for the model. Paste it into a Claude Code session in this repo and the ticket gets written. Nothing is sent anywhere; the manual only copies text to your clipboard.

The queue check.

The note modal lists the tickets already queued for the section you are annotating, with the rest of the queue one click deeper. Recognize your request in the list, mark it as yours, and the payload files your note as an addition: the existing ticket updates in place, its requirements absorb the new information, and no twin is created.

TICKET-0004 idea needs answers
Let the queue filter by ticket type.
✓ Filing as an addition to this ticket
Example: the queue check mid-note, after the reader recognized their request

After you paste.

The session grounds the note first: it reads this manual's section, the code behind it, and the project's planning docs, then writes a ticket with user stories, functional requirements, and acceptance criteria. A completeness gate decides whether the ticket is ready to build or needs answers first; a thin note is never padded into false precision.

Tickets land in docs/tickets/, and the queue index in this file updates so the next reader's queue check sees them.

Roadmap previews.

A feature with planned changes is a moving target, and feedback on its current shape is often feedback on something already being replaced. Previews fix the aim.

A icon sits wherever a plan exists. Click it (the one in this sentence works) to see the planned state, its status, and the roadmap source it comes from.

The scope rule follows. A note on a previewed item files against the plan, not the current behavior; the note modal says so and scopes the payload accordingly. An icon inside a heading locks that whole section to its plan. An icon inline in a sentence offers its item as an optional target.

This repo has no roadmapped work today, so the icon above opens a demonstration and doesn't rescope notes on this section.

The concept glossary.

Product-novel terms carry a dotted underline and a hover definition the first time they appear. You have met several already in this manual.

The discipline: one alert per concept, on its first instance only, and only when that instance falls before the passage that defines the concept in full. Once the manual has explained a term, the reminders stop. A document that keeps footnoting what you already read is noise.

Each entry records where its definition lives, so when prose moves during an update, the alert boundary moves with it.

The ticket skill.

/living-manual:ticket turns a pasted note, or a free-form report, into a developer ticket in docs/tickets/. One ticket per submission, unless it clearly contains independent requests.

Grounded before written.

Every claim about current behavior must be verifiable in code or docs; the skill reads the named section and the code behind it before writing a word. Classification is honest: designed-but-frustrating behavior is filed as feedback naming the governing decision, not as a bug, and a mislabeled submission is reclassified with a line saying so.

The completeness gate.

A ticket is marked ready only when every functional requirement is unambiguous and testable, and the acceptance criteria cover all of them. Anything resting on a guess flips it to needs answers, with open questions that carry their own resolution: each names the requirement it blocks, proposes a default, and says what changes if the default is wrong. Answer them all and the ticket flips to ready.

What a ticket contains.

  • A summary, and the submission verbatim as the origin record.
  • Current behavior (or planned behavior, for plan-revision tickets) with file references.
  • User stories; every requirement traces to one.
  • Numbered functional requirements, each testable.
  • Given/When/Then acceptance criteria covering every requirement.
  • Open questions, when the gate demands them.

Jira.

With the Atlassian MCP configured, each ticket also becomes an issue in the mapped Jira project, and additions update the same issue rather than creating twins. A Jira failure never blocks the local ticket.

This repo has Jira sync disabled: no Atlassian MCP was available at setup. Install the MCP and re-run setup to enable it.

Setup and updates.

First run.

Install the plugin (claude --plugin-dir /path/to/living-manual, or add the repo as a marketplace) and run /living-manual:manual in your repo. Setup walks six steps, each landing on disk, so an interrupted run resumes where it stopped:

  1. Orientation: the codebase is inventoried and the full feature map is printed in the reply for you to read. You confirm or correct what's on screen before a word of the manual is written.
  2. Brand: palette, type, and voice are read from your brand assets. Without any, a neutral palette stands in and is noted for a later pass.
  3. Jira: optional, and only offered when the Atlassian MCP is present.
  4. Config: .living-manual.json at the repo root records all of it.
  5. Wiring: the pre-push guard installs, CLAUDE.md gains the update instruction, and the tickets directory is created.
  6. Build: the shell is scaffolded, the prose written, and the result verified before it's reported done.

Every run after.

/living-manual:manual update diffs reality against the manual: it lists the user-facing commits since the base marker, revises only the sections those changes touch, appends a dated entry to What's new, and re-stamps the marker. A manual already matching the code says so and stops.

Shipped roadmap items lose their previews; the content moves into the section and What's new. New concepts enter the glossary with their defining passage recorded.

The config file.

.living-manual.json holds the product identity, paths, brand, and Jira settings. The user_facing_paths list decides which commits count against freshness: changes elsewhere (CI, tests, internal tooling) never block a push. The full key list is in the reference below.

The staleness guard.

The manual records the commit it last matched: the base marker, stamped in a comment near the top of this file and read by the scripts. The manual is stale when user-facing commits postdate that marker.

Enforcement is two-layered. CLAUDE.md instructs every session in the repo to run the update flow before any push, so freshness is part of the normal workflow. The pre-push hook is the backstop: a push with user-facing commits past the marker is blocked, with a message naming the fix.

The escape hatch costs the next reader. git push --no-verify bypasses the guard once. The push succeeds, and the next dev opens a manual describing an app that no longer exists. Fix the manual instead; the update flow only revises what changed.

The guard reads user_facing_paths from the config, so documentation-only and tooling-only pushes pass untouched.

Reference.

Config keys.

KeyMeaning
product, tagline, tagline_pill Identity shown in the manual's hero.
manual_path Where the manual lives. Default docs/USER_MANUAL.html.
tickets_dir The ticket queue. Default docs/tickets.
ticket_skill The command the note payload invokes.
user_facing_paths Path prefixes whose commits make the manual stale.
brand ink, surface, deep, accent, warn, caution, font stack, hero gradient, logo data URI. Missing keys fall back to a neutral palette.
jira enabled, project key, and the note-type to issue-type mapping.

Scripts.

The scripts do the mechanical work so model time goes to content and judgment. All run from the plugin; nothing needs installing.

ScriptDoes
state.sh One JSON line: configured or not, manual base, HEAD, wiring.
inventory.sh Codebase orientation for setup: framework signals, surfaces, docs, brand candidates.
stale.sh CURRENT, or the commits and user-facing files since the base marker.
install-hook.sh Installs the pre-push guard.
scaffold.py Instantiates the manual shell from the brand config.
sync-index.py Rewrites the manual's data blocks from a JSON payload.
tickets-index.py Rebuilds the queue index from the ticket files, so the index can't drift.
next-ticket.sh The next free ticket number.
verify.py Static integrity checks; prints OK or findings.

Ticket statuses.

StatusMeaning
ready Complete: requirements testable, criteria cover them, no guesses.
needs answers Open questions block completeness; answering them all flips the ticket to ready.
shipped Landed in a release. Dropped from the queue index, kept as the record.
declined Closed without building. Kept as the record.

Requires git and python3. With node present, verification also syntax-checks the manual's script.
